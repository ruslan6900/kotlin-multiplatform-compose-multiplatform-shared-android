## Migration Guidelines: как переносить эти наработки в основной проект

Этот PoC создавался не только ради самого PoC, а как подготовка к более серьёзной задаче: понять, как переносить существующее приложение в сторону `Kotlin Multiplatform + Compose Multiplatform + Aurora`.

Поэтому главный практический вопрос звучит так:

- что из этого можно переносить почти напрямую;
- что нужно адаптировать;
- что пока лучше не трогать;
- и в каком порядке это делать, чтобы не сломать рабочий Android-проект.

Ниже описан наиболее реалистичный путь миграции.

## 1. Что можно переносить в первую очередь

В первую очередь в `commonMain` имеет смысл переносить то, что уже хорошо живёт без жёсткой привязки к Android framework.

Обычно это:

- модели данных;
- DTO и сериализация;
- repository/use-case слой;
- сетевой API через Ktor;
- DI-модули через Koin;
- экранные `ViewModel`, если они не завязаны напрямую на Android-only API;
- общие composable-экраны;
- state-driven навигационную логику;
- persistence abstraction;
- часть persistence-реализации, если она уже может жить на Room KMP.

Это даёт максимальную пользу при минимальном риске, потому что именно здесь обычно лежит основной переиспользуемый код.

## 2. Что не нужно переносить в `commonMain` насильно

Одна из самых частых ошибок при миграции - пытаться унести в общий модуль вообще всё подряд.

Не стоит насильно переносить в `commonMain`:

- Android `Context`;
- `Activity`;
- Android services;
- Android permission flow;
- platform filesystem specifics;
- platform notification APIs;
- platform VPN/integration APIs;
- всё, что жёстко привязано к конкретному SDK или native runtime.

Для таких вещей лучше сразу проектировать тонкий platform bridge.

Правильная цель не "убрать весь платформенный код", а "оставить платформенный код только там, где он действительно нужен".

## 3. Рекомендуемый порядок миграции

На практике safest path обычно такой.

### Шаг 1. Выделить data/model слой

Сначала вынести в shared-код:

- модели;
- сериализацию;
- базовые repository interfaces;
- Ktor API-клиент;
- общие utility-классы;
- общие state-модели.

Почему так:

- этот слой обычно проще всего изолировать;
- он быстро показывает, насколько проект вообще готов к KMP;
- здесь меньше UI/runtime-рисков.

### Шаг 2. Вынести DI

После этого перенести:

- Koin modules;
- graph registration;
- shared construction логики для repositories/services/storage.

Почему это полезно:

- DI хорошо показывает границы между shared и platform code;
- после этого экранный код легче переключать на shared зависимости.

### Шаг 3. Вынести сетевой слой на Ktor

Если в Android-проекте сеть ещё не на Ktor, следующим сильным шагом будет унифицировать network API.

Цель:

- общий клиент;
- общие DTO;
- общий parsing;
- платформенные только engine.

Именно такой паттерн лучше всего масштабируется на Android + iOS + Aurora.

### Шаг 4. Подготовить persistence abstraction

Даже если persistence пока остаётся платформенной, полезно заранее ввести интерфейсы:

- storage interfaces;
- repository interfaces;
- mapping слой между UI и БД.

Это позволяет потом заменить backend без переписывания экранов.

### Шаг 5. Переносить UI экраны по одному

Не нужно пытаться в один шаг перенести весь UI.

Лучше идти так:

1. один экран;
2. потом второй;
3. потом навигация между ними;
4. потом общий root screen;
5. потом более сложные интерактивные сценарии.

Это особенно важно, если целевая платформа - Aurora, где runtime-стабильность нужно проверять очень рано.

## 4. Как переносить Compose UI

Если проект уже на Jetpack Compose, это сильное преимущество.

Обычно миграция идёт по такому принципу:

- сначала отделяется UI state от Android-specific зависимостей;
- затем composable-функции переводятся на shared dependencies;
- затем навигационная логика упрощается до shared state-driven модели или аккуратно адаптируется;
- затем root screen начинает жить в `commonMain`.

Очень важно:

- сначала переносить простые экраны;
- избегать слишком сложных platform integration точек в самом начале;
- не полагаться на то, что всё автоматически заработает на Aurora так же, как на Android.

По итогам этого PoC можно считать реалистичным такой подход:

- shared composable-элементы использовать можно;
- shared screen stack использовать можно;
- но для Aurora нужно держать отдельный диагностический сценарий на ранней стадии миграции.

## 5. Как переносить Room

Для текущего состояния проекта Room уже стал единым persistence backend, и это важный сигнал для миграции.

Практический подход такой:

1. сначала описать entities и DAO так, чтобы они не зависели от Android UI слоя;
2. вынести database API в shared-уровень;
3. вынести storage abstraction;
4. отдельно решить platform-specific database path;
5. проверить open/create/save/read сценарии на каждой платформе.

Особенно важно не смешивать:

- shared DB contract;
- platform-specific способ разместить файл БД.

В реальном проекте это означает, что Room можно переносить постепенно, без необходимости ломать экранный слой.

## 6. Можно ли оставить разный persistence backend на разных платформах

Технически да, можно.

Например:

- Android/iOS могут временно жить на одном backend;
- Aurora - на другом;
- всё это может прятаться за общим `PostsStorage`-подобным интерфейсом.

Но по результатам этого PoC более здоровая стратегия - стремиться к единому backend там, где это реально возможно.

Почему:

- меньше divergence в поведении;
- меньше платформенных багов второго порядка;
- проще тестировать;
- проще сопровождать persistence-логику;
- легче переносить фичи между платформами.

То есть "разные backend на разных платформах" - допустимый переходный этап, но не лучший конечный результат.

Текущее состояние этого PoC уже показывает более сильный вариант:

- один общий Room-based persistence contract;
- Android на Room;
- iOS на Room;
- Desktop на Room;
- Aurora на Room.

Это не означает, что так обязано быть в любом проекте, но показывает, что для данного стека единый backend уже реалистичен, а не только теоретически возможен.

## 7. Что делать специально для Aurora

При переносе в основной проект для Aurora стоит сразу заложить отдельный рабочий пакет задач:

- отдельный Aurora build variant;
- отдельные совместимые зависимости;
- отдельный build/install/run pipeline;
- отдельные IDE run-конфигурации;
- отдельный render smoke screen;
- отдельное логирование стартового сценария.

Это нужно не потому, что Aurora "плохая", а потому что в ней нужно раньше видеть границу между:

- проблемой бизнес-логики;
- проблемой shared UI;
- проблемой runtime;
- проблемой launcher/emulator.

Если этого не сделать, диагностика станет сильно дороже.

## 8. Как не сломать существующий Android-проект

Самое важное правило миграции: не переписывать всё одним большим коммитом.

Лучше идти маленькими безопасными этапами:

1. выделить новый shared module или новый shared source set;
2. перенести одну вертикаль функциональности;
3. собрать Android;
4. проверить regression;
5. только потом переносить следующий слой.

Хороший принцип:

- Android должен продолжать собираться после каждого шага;
- shared-слой должен добавляться инкрементально;
- Aurora target нужно подключать рано, но не пытаться сразу доводить до идеального production UX.

## 9. Рекомендуемая стратегия для основного проекта

Если переносить эти идеи в существующий проект, я бы рекомендовал такой порядок:

1. определить, какие экраны и use-case логика наименее завязаны на Android;
2. вынести network/models/serialization в shared;
3. подключить Koin в shared-слой;
4. вынести часть UI в shared Compose;
5. ввести Room KMP или хотя бы shared persistence abstraction;
6. подключить Aurora build variant;
7. сделать минимальный Aurora smoke dashboard;
8. только потом переносить более сложные фичи.

Это наиболее безопасный путь, потому что он даёт раннюю проверку жизнеспособности Aurora-target, но не заставляет сразу переносить весь проект целиком.

## 10. Что этот PoC советует делать, а чего не советует

### PoC советует делать

- переносить shared state и shared UI постепенно;
- держать платформенные адаптеры тонкими;
- заранее отделять abstraction от platform implementation;
- проверять Aurora runtime как можно раньше;
- иметь отдельный smoke screen для диагностики;
- документировать реальные рабочие run/build/install сценарии.

### PoC не советует делать

- переносить весь проект в KMP одним шагом;
- завязывать shared-слой на Android-only API;
- смешивать runtime workaround с общей бизнес-логикой;
- надеяться, что Aurora сама будет вести себя как Android;
- принимать успешную сборку за признак готовности runtime.

## 11. Краткий итог миграции

По результатам этого проекта можно сделать практический вывод:

- перенос общего UI, DI, сети и Room в shared-слой реалистичен;
- Android и Aurora действительно могут использовать один и тот же значимый объём кода;
- главный риск миграции находится не в моделях и не в repository-слое, а в platform/runtime integration;
- Aurora нужно подключать рано и проверять отдельно;
- лучший путь - инкрементальная миграция с постоянной проверкой Android и Aurora после каждого шага.

Именно так этот PoC и стоит использовать: не как шаблон "скопировать один в один", а как дорожную карту для аккуратного переноса существующего приложения на KMP + Compose Multiplatform + Aurora OS.
