## Как общий Compose UI вообще запускается на Aurora

Это один из самых важных вопросов во всём PoC.

На первый взгляд может казаться, что Aurora OS слишком далека от обычного Android-механизма, и поэтому общий Compose UI из `commonMain` не должен там работать.

На практике оказалось иначе:

- общий Compose-код действительно может быть собран один раз как shared UI-слой;
- этот shared UI действительно может выполняться на Aurora;
- но между shared composable-кодом и реальным окном Aurora всё равно есть платформенный bridge.

Чтобы понимать происходящее правильно, полезно разложить это по слоям.

## 1. Что именно находится в `commonMain`

В `commonMain` лежит не "готовое окно приложения", а декларативное описание UI и логики.

То есть там находятся:

- `@Composable`-функции;
- screen state;
- `ViewModel`;
- действия пользователя;
- состояния network/database;
- layout-структуры вроде `Column`, `Row`, `Box`, `Button`, `Text`;
- общая навигационная логика между экранами;
- общий слой данных, который кормит этот UI.

Это не Android XML и не Aurora native widget tree.

Это Kotlin-код, который описывает:

- что должно быть показано;
- как UI реагирует на state;
- как state меняется при действиях пользователя.

## 2. Почему этот код можно шарить между платформами

Compose Multiplatform устроен так, что `@Composable`-код в `commonMain` не привязан напрямую к одной конкретной ОС.

Он опирается на кроссплатформенный Compose runtime:

- runtime отслеживает состояние;
- runtime запускает recomposition;
- runtime строит внутреннее представление UI;
- platform backend рисует это в конкретной среде выполнения.

Именно это и позволяет одним и тем же composable-функциям работать не только на Android, но и на других target.

То есть shared UI работает не потому, что Aurora "понимает Android UI", а потому что Compose сам предоставляет более высокий уровень абстракции.

## 3. Где здесь роль Aurora

Aurora не исполняет `@Composable`-код напрямую "магическим" образом.

Нужна цепочка:

1. Kotlin Multiplatform target собирает общий код вместе с Aurora-specific target;
2. Compose runtime и связанный backend подключаются в Aurora-compatible варианте;
3. Aurora toolchain собирает приложение в нужный бинарный/пакетный формат;
4. приложение стартует в Aurora runtime;
5. platform layer создаёт окно и связывает его с Compose rendering pipeline;
6. уже после этого shared composable-код начинает реально рисоваться.

То есть Aurora здесь выступает как host environment для исполнения KMP/Compose target, а не как отдельный UI toolkit, в который мы вручную переписываем все экраны.

## 4. Почему один и тот же Composable может отображаться на Android и Aurora

Потому что сам composable описывает не платформенный widget, а UI-функцию.

Например, когда мы пишем:

- `Text(...)`
- `Button(...)`
- `Column(...)`
- `Box(...)`

мы описываем UI через Compose abstraction layer.

Дальше эта abstraction layer уже превращается в конкретный rendering behavior платформы.

В результате:

- Android получает свой runtime path;
- Desktop получает свой runtime path;
- Aurora получает свой runtime path;
- а `commonMain` код остаётся тем же самым.

Именно поэтому для пользователя проекта это выглядит как "одни и те же composable-элементы работают и там, и там".

## 5. Почему для этого не нужен Android Navigation

Важно отдельно проговорить одну вещь: этот PoC не опирается на `androidx.navigation-compose`.

Навигация здесь сделана как shared state-driven переключение экранов.

Это значит:

- выбран текущий `AppScreen`;
- UI в `commonMain` показывает нужный экран в зависимости от этого состояния;
- переключение между экранами тоже происходит в shared-коде.

Почему это важно:

- такая навигация не завязана на Android framework;
- её проще перенести на Aurora;
- это хороший способ доказать, что многосценарный UI может жить в общем модуле и без Android-only navigation stack.

То есть здесь общий не только внешний вид, но и сам способ перехода между экранами.

## 6. Как Compose state и recomposition работают на Aurora

Когда приложение запускается на Aurora, shared Compose runtime по тем же базовым принципам, что и на других target, делает следующее:

- читает текущий state;
- строит композицию;
- подписывается на observable state;
- перерисовывает нужные части UI при изменении state.

Поэтому на Aurora в PoC удалось показать:

- обновление времени каждую секунду через `Flow`;
- смену статусов после Ktor-запроса;
- смену статусов после работы с Room;
- обновление лог-панели;
- отображение сохранённой заметки после чтения из БД.

Это важный сигнал: на Aurora работает не просто статическая картинка, а настоящий reactive Compose UI.

## 7. Тогда почему вообще возникали проблемы с белым или красным экраном

Потому что возможность "теоретически исполнить shared Compose UI" и стабильность конкретного runtime rendering path - это не одно и то же.

В ходе PoC стало видно:

- базовые Compose-примитивы действительно рисуются;
- значимая часть сложного UI тоже рисуется;
- Ktor и Room могут обновлять shared state;
- но жизненный цикл повторного запуска на Aurora ведёт себя нестабильно.

Именно из-за этого и понадобился отдельный render smoke dashboard.

То есть проблема оказалась не в самой идее общего Compose UI, а в runtime-стабильности конкретного сценария запуска Aurora-приложения.

## 8. Что в этом механизме реально является общим

В практическом смысле общими здесь являются:

- composable-функции;
- модели UI-состояния;
- обработчики действий;
- shared repositories;
- shared `ViewModel`;
- shared Room API;
- shared Ktor API;
- shared time ticker;
- shared логика переключения экранов.

Именно поэтому Android и Aurora могут показывать один и тот же функциональный UI-сценарий без переписывания экранов по отдельности.

## 9. Что всё равно остаётся платформенным

Несмотря на общий UI, остаются платформенными:

- entry point приложения;
- создание runtime window;
- привязка к platform lifecycle;
- network engine;
- database file path;
- сборка и packaging;
- emulator install/run pipeline;
- часть Aurora-specific стабилизации запуска.

Это не противоречие, а нормальное устройство Compose Multiplatform.

Shared UI не означает отсутствие платформенной инфраструктуры. Он означает, что бизнес-ценная часть UI больше не пишется заново для каждой ОС.

## 10. Что доказал этот PoC

Этот проект доказал не в теории, а практически следующее:

- `commonMain` может содержать не только utility-код, но и реальный UI;
- Compose UI из общего модуля действительно может запускаться на Aurora;
- один и тот же shared screen stack может использоваться на Android и Aurora;
- общий state/reactive подход тоже сохраняется;
- платформенный код можно ужать до слоя интеграции и инфраструктуры.

Главная оговорка при этом остаётся такой:

- shared Compose UI на Aurora возможен;
- но стабильность конкретного runtime path нужно проверять отдельно и внимательно, особенно на повторных запусках и при усложнении UI.

Именно поэтому этот PoC сочетает в себе сразу две роли:

- демонстрацию общего Compose-стека;
- диагностический стенд для проверки реального поведения Aurora runtime.
