# Aurora KMP Demo

PoC-проект для проверки, насколько далеко можно увести общий стек `Kotlin Multiplatform + Compose Multiplatform` в сторону `Android + iOS + Aurora OS`, сохраняя максимум логики и UI в `commonMain`.

Проект не является production-образцом. Это исследовательский стенд, в котором специально проверяются:

- общий Compose UI;
- общий DI через Koin;
- общий сетевой слой через Ktor;
- общий persistence-слой через Room;
- граница между `commonMain` и платформенными `actual`-реализациями;
- особенности запуска Compose-приложения на Aurora OS;
- возможность сборки, упаковки и запуска общего KMP-приложения в Aurora emulator.

## Цель PoC

Главная цель этого проекта не просто "собрать что-то под Aurora", а проверить практический сценарий:

1. можно ли держать основную бизнес-логику и UI в `commonMain`;
2. можно ли использовать те же Compose-компоненты, что и на Android/iOS/Desktop;
3. можно ли оставить общими DI, сеть, модели данных и persistence API;
4. где именно начинаются реальные платформенные ограничения Aurora;
5. какой объём отдельного Aurora-кода всё-таки необходим.

Иными словами, это не демо красивого приложения, а инженерная проверка переносимости общего KMP-стека на Aurora OS.

## Что сейчас умеет проект

На текущем этапе PoC уже умеет:

- собираться как `main`-вариант для Android/Desktop/iOS;
- собираться как `aurora`-вариант через отдельный Aurora Gradle pipeline;
- создавать RPM-пакеты для Aurora `aarch64` и `x86_64`;
- устанавливаться в Aurora emulator;
- запускаться в Aurora emulator;
- отображать диагностический Compose-экран для Aurora;
- выполнять HTTP-запрос через Ktor;
- сохранять данные в Room;
- сохранять и восстанавливать пользовательский текст из Room;
- показывать текущее время с обновлением раз в секунду.

## Почему проект разделён на `main` и `aurora`

В репозитории сознательно существуют два build-варианта:

- `main`
- `aurora`

Это сделано потому, что Aurora использует отдельные:

- Gradle plugins;
- runtime-артефакты;
- packaging rules;
- device/emulator deployment tasks;
- некоторые версии библиотек с aurora-суффиксами.

То есть с точки зрения исходников это один KMP-проект, а с точки зрения toolchain и packaging это два разных способа сборки одного и того же приложения.

## Важная оговорка

Хотя в проекте используется общий Compose UI, это не означает, что любая Compose-фича автоматически и одинаково стабильно работает на Aurora.

Во время PoC были обнаружены реальные особенности:

- потребовался отдельный Aurora-friendly smoke dashboard;
- пришлось учитывать проблемы повторного запуска UI;
- пришлось отдельно настраивать storage path для Room на Aurora;
- пришлось делать Aurora-specific build/deploy/run pipeline.

Именно поэтому этот PoC полезен: он показывает не рекламную картину, а реальные границы переносимости.
