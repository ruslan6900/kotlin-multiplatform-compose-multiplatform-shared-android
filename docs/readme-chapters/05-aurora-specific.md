## Что пришлось писать специально для Aurora

Это PoC важен не только тем, что он запускает общий Compose UI, но и тем, что показывает реальную цену выхода на Aurora OS.

Если смотреть честно, Aurora здесь не просто "ещё одна платформа из коробки". Для неё пришлось делать отдельный слой адаптации, отдельную сборочную конфигурацию и отдельные runtime-решения.

Ниже описано, что именно оказалось Aurora-specific и почему это нельзя было просто целиком спрятать в `commonMain`.

## 1. Отдельный build variant и отдельный Gradle script

Для Aurora в проекте используется отдельный build variant и отдельный Gradle script.

Смысл этого разделения в том, что Aurora-сборка не совпадает один в один с обычной Android/JVM/KMP-конфигурацией.

Для Aurora понадобились:

- отдельные плагины Aurora toolchain;
- отдельные совместимые BOM и версии зависимостей;
- отдельные build/install/run task;
- упаковка в RPM;
- привязка к Aurora emulator/runtime.

Почему это нельзя честно перенести в `commonMain`:

- `commonMain` предназначен для исходного кода, а не для описания платформенной toolchain;
- build logic и deployment pipeline живут на уровне Gradle, а не на уровне shared Kotlin-кода.

## 2. Aurora-compatible зависимости

Для Aurora пришлось использовать отдельные совместимые артефакты и BOM:

- `koin-bom:4.2.0-aurora`
- `ktor-bom:3.4.2-aurora`
- `kotlinx-datetime:0.7.1-aurora`
- `ktor-client-curl`

Почему это важно:

- Aurora runtime не всегда совместим с теми же бинарными наборами, что обычный JVM/Linux или Android;
- одна и та же библиотека логически "та же самая", но practically для Aurora может требоваться свой вариант публикации.

Почему это нельзя решить только общим модулем:

- `commonMain` не выбирает бинарные артефакты публикации;
- это решается через platform-specific dependency configuration.

## 3. Aurora-specific запуск и диагностика рендера

На практике одна из самых важных особенностей PoC проявилась не на уровне компиляции, а на уровне запуска.

Было подтверждено:

- приложение собирается;
- RPM создаётся;
- RPM ставится в emulator;
- окно приложения открывается;
- но полноценный shared UI на Aurora в обычном режиме отображается нестабильно.

Именно поэтому в проект был добавлен отдельный Aurora-friendly render smoke path.

Он нужен для того, чтобы:

- проверить, что окно вообще живо;
- убедиться, что Compose может рисовать фон, текст, кнопки, блоки;
- диагностировать, в какой момент более сложный UI перестаёт стабильно отображаться.

Почему это нельзя полностью спрятать в общий UI:

- проблема связана не только с composable-кодом, но и с поведением конкретного Aurora runtime;
- нужно было иметь специальный диагностический сценарий именно для Aurora emulator;
- общий код не должен насильно переводить Android/iOS в такой же "аварийный" режим.

## 4. Aurora-friendly bootstrap

В проекте понадобился более осторожный bootstrap при старте Aurora UI.

Смысл был в том, чтобы:

- контролировать момент инициализации Koin;
- логировать запуск;
- видеть идентификатор конкретного старта;
- иметь визуальный smoke entry screen;
- уменьшить количество moving parts до того, как UI гарантированно отрисуется.

Почему это нельзя на 100% сделать одинаково для всех платформ:

- Android и Desktop стартуют заметно стабильнее;
- проблема была найдена именно в Aurora runtime-сценарии;
- одинаковое поведение для всех платформ здесь бы только ухудшило архитектуру и усложнило normal path.

## 5. Работа с файловой системой Aurora

Хотя Room сейчас используется как единый persistence backend, физический путь к базе и логам на Aurora всё равно определяется через platform-specific слой.

Это нужно потому, что на Aurora:

- приложение живёт в sandbox;
- storage path отличается от Android/iOS/Desktop;
- диагностику иногда удобно писать в app-local filesystem location.

Почему это нельзя реализовать целиком в `commonMain`:

- `commonMain` не знает, где именно на конкретной ОС должна лежать база;
- это часть окружения, а не часть общей бизнес-логики.

## 6. Aurora run/install pipeline

Для Aurora пришлось отдельно учитывать:

- RPM packaging;
- установку на emulator;
- повторный запуск;
- kill/reinstall сценарии;
- диагностику поведения после повторного открытия из GUI.

Это особенно важно, потому что в ходе PoC был найден нетривиальный runtime-эффект:

- после "чистого" запуска через Gradle UI мог нормально отрисоваться;
- после закрытия и повторного открытия из самой Aurora OS приложение иногда показывало только красный фон.

То есть проблема оказалась уже не в "собирается / не собирается", а в жизненном цикле Aurora runtime.

Почему это нельзя решить просто в общем модуле:

- `commonMain` не управляет `runtime-manager-tool`, emulator lifecycle, launcher policy и reinstall/run pipeline;
- это внешняя инфраструктура платформы.

## 7. Почему нельзя всё Aurora-специфичное просто унести в общий код

Здесь важно не путать два разных слоя:

1. общий код приложения;
2. интеграция приложения с платформой.

`commonMain` отлично подходит для:

- UI;
- state;
- use-case логики;
- репозиториев;
- сетевого API;
- значительной части persistence.

Но `commonMain` не предназначен для:

- выбора platform engine;
- filesystem sandbox specifics;
- build plugins;
- RPM packaging;
- запуска emulator tasks;
- интеграции с Aurora runtime-manager.

Если пытаться затолкать это в shared-модуль, архитектура станет не более кроссплатформенной, а менее честной и менее поддерживаемой.

## 8. Что это означает для реального проекта

Для production-проекта под Android + iOS + Aurora отсюда следует здоровая стратегия:

- максимум UI, domain и data logic хранить в `commonMain`;
- минимальный платформенный слой держать тонким;
- Aurora-specific build/deploy/runtime решения изолировать отдельно;
- все спорные runtime workaround фиксировать явно, а не размазывать по общему коду.

Именно такой подход в этом PoC и получился:

- общий код отвечает за функциональность приложения;
- Aurora-specific код отвечает за совместимость, запуск и диагностику поведения на Aurora OS.

Это и есть наиболее практичная модель переноса существующего KMP/Compose проекта на Aurora.
