## Что осталось платформозависимым

Одна из главных целей этого PoC была не просто "запустить KMP", а понять реальную границу между тем, что можно унести в `commonMain`, и тем, что всё равно придётся писать отдельно для платформ.

Итог получился практический:

- очень большая часть UI и бизнес-логики действительно живёт в `commonMain`;
- но runtime-entry, filesystem, network engine, database builder и Aurora build/deploy pipeline всё равно остаются платформозависимыми.

Ниже это разложено по слоям.

## 1. Что удалось оставить в `commonMain`

В общий модуль сейчас вынесены:

- Compose UI экраны;
- UI-state;
- экранная логика;
- простая навигация между экранами;
- Koin-модули;
- `ViewModel`;
- Ktor repository/service layer;
- модели сериализации;
- поток времени через `Flow`;
- Room DAO, entities и общая database API;
- `PostsStorage` и его общая Room-реализация;
- логика сохранения и чтения заметки;
- диагностические shared-компоненты для Aurora render smoke dashboard.

Это означает, что основное поведение приложения не дублируется между Android, iOS и Aurora.

На практике это самое важное доказательство PoC: shared-слой здесь не декоративный, а реально рабочий.

## 2. Что осталось платформенным в UI-слое

Хотя сами composable-функции живут в `commonMain`, точка входа в приложение у каждой платформы своя.

Например:

- Android использует `Activity`;
- iOS использует свой native entry point;
- Desktop использует desktop launcher;
- Aurora использует свой target/runtime launcher.

То есть `App()` общий, но способ "встроить" его в окно и runtime у каждой платформы свой.

Именно поэтому платформенные source set-ы всё равно нужны, даже если почти весь UI общий.

## 3. Что осталось платформенным в network-слое

Общий код знает только про абстракцию клиента и про то, как выполнять запросы.

Но конкретный HTTP engine выбирается отдельно:

- Android: `OkHttp`
- iOS: `Darwin`
- Desktop: `Java`
- Aurora: `Curl`

Это связано с тем, что `ktor-client-core` сам по себе не умеет выполнять реальные запросы без engine.

То есть network API можно полностью унифицировать, а transport backend всё равно остаётся платформенным.

## 4. Что осталось платформенным в persistence-слое

Сам persistence API сейчас во многом общий:

- entities;
- DAO;
- база;
- storage abstraction;
- бизнес-логика сохранения и чтения.

Но путь к файлу базы и способ получить filesystem location всё равно платформозависимы.

Причина простая:

- у Android один механизм доступа к app storage;
- у iOS другой sandbox path;
- у Desktop обычный JVM filesystem;
- у Aurora свой app sandbox и свои директории.

В текущем состоянии PoC это выражается так:

- Android: app database directory через `Context.getDatabasePath(...)`;
- iOS: `~/Library/Application Support/aurora-kmp-demo/posts-room.db` внутри sandbox;
- Desktop: `~/.aurora-kmp-demo/posts-room.db`;
- Aurora: `~/Documents/aurora-kmp-demo/posts-room.db`.

Поэтому общий код не должен сам угадывать, куда положить файл базы. Он получает это через platform-specific слой.

## 5. Что осталось платформенным в DI/bootstrap-слое

Сами Koin-модули общие, но момент инициализации DI зависит от платформы и от способа запуска приложения.

Особенно это стало заметно на Aurora:

- обычный сценарий запуска общего UI оказался недостаточно надёжным;
- понадобился Aurora-friendly bootstrap;
- пришлось отдельно следить за тем, что инициализировалось до первого реального рендера.

Отсюда важный вывод:

- контейнер зависимостей может быть общим;
- но lifecycle его старта не всегда можно оставить полностью универсальным.

## 6. Что осталось платформенным в build/deploy

Сборка Android и сборка Aurora принципиально различаются.

Android использует привычный Android Gradle pipeline.

Aurora использует:

- отдельный build variant;
- отдельный Gradle script;
- Aurora build plugins;
- RPM packaging;
- установку в emulator через Aurora tooling;
- специализированные run/install tasks.

Это не недостаток архитектуры проекта, а нормальная реальность: deployment-пайплайн почти всегда платформозависим даже при очень хорошем shared-коде.

## 7. Почему это нормально

Иногда при первом знакомстве с KMP кажется, что "кроссплатформенный" проект должен быть почти полностью без platform-specific кода.

На практике это неверное ожидание.

Правильная цель обычно такая:

- общий domain;
- общий data layer;
- общий UI, если это Compose Multiplatform;
- тонкий платформенный runtime bridge.

Именно это здесь и сделано.

Платформенный код не исчезает совсем, но он становится:

- заметно меньше;
- проще;
- ближе к инфраструктуре, чем к бизнес-логике;
- легче сопровождать.

## 8. Практический вывод по проекту

Для этого PoC граница получилась примерно такой:

Полностью или почти полностью общие:

- UI;
- навигационная логика;
- DI;
- сетевой слой;
- модели;
- значительная часть persistence;
- экранные `ViewModel`.

Неизбежно платформенные:

- entry point;
- HTTP engine;
- filesystem path;
- некоторые части database setup;
- build/deploy/install/run tooling;
- Aurora runtime-specific стабилизация запуска.

Это хороший результат для проекта, который одновременно целится в Android, iOS и Aurora OS.
