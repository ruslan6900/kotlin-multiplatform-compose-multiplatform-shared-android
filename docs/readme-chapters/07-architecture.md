## Архитектура проекта

Этот проект лучше всего воспринимать не как "набор экранов", а как небольшой исследовательский KMP-стенд с многослойной архитектурой.

Задача архитектуры здесь была не в академической чистоте, а в практической проверке:

- насколько глубоко можно унести код в `commonMain`;
- где реально заканчивается shared-слой;
- как Compose UI, DI, network и persistence ведут себя на Android, iOS, Desktop и Aurora.

При этом архитектура всё равно получилась достаточно чистой и хорошо читаемой.

## 1. Общая схема слоёв

Проект можно мысленно разделить на пять слоёв:

1. platform/runtime layer
2. shared app bootstrap layer
3. presentation layer
4. data layer
5. infrastructure/build layer

В более прикладном виде это выглядит так:

```text
Platform entry points
    ->
App bootstrap
    ->
Shared Compose UI + ViewModel
    ->
Repositories / storage / services
    ->
Ktor / Room / platform actuals
```

## 2. Platform/runtime layer

Это самый внешний слой.

Он отвечает за то, чтобы приложение вообще стартовало на конкретной платформе.

Сюда входят:

- Android entry point;
- iOS entry point;
- Desktop launcher;
- Aurora target/runtime entry;
- platform-specific HTTP engine;
- platform-specific filesystem/database path;
- platform-specific build/deploy behavior.

Этот слой специально держится тонким.

Его задача не "содержать логику приложения", а только:

- создать runtime-контекст;
- подключить shared `App()`;
- передать платформенные зависимости в общий слой.

## 3. Shared app bootstrap layer

Следующий слой - это общий старт приложения.

Он связывает:

- Compose root;
- Koin;
- общий app state;
- стартовые screen mode и diagnostic mode;
- Aurora-friendly boot path.

Именно здесь решается:

- какой root UI показывать;
- запускать ли diagnostic smoke screen;
- запускать ли normal demo screen flow;
- инициализирован ли DI;
- какие shared зависимости должны быть доступны экрану.

Это важный слой, потому что он уже не чисто платформенный, но ещё и не конкретный экран.

## 4. Presentation layer

Это центральный слой PoC.

Сюда входят:

- `App()`;
- root `NavHost` для `SuperSmoke -> RenderSmoke -> NormalDemo`;
- nested `NavHost` для `Home -> Network -> Database`;
- shared composable-экраны;
- Aurora diagnostic composables;
- screen state;
- user interactions;
- `ViewModel`;
- логика показа статусов, логов, текста, кнопок и результатов операций.

В проекте presentation layer в основном живёт в `commonMain`.

Это означает, что:

- UI действительно шарится;
- Android и Aurora используют один и тот же базовый экранный код;
- общая логика отображения не дублируется по платформам;
- переходы между экранами тоже не дублируются по платформам.

### Что есть внутри presentation layer

Основные части этого слоя:

- `HomeScreen`
- `NetworkScreen`
- `DatabaseScreen`
- диагностический Aurora dashboard
- render smoke composables
- navigation graph
- screen `ViewModel`

Отдельно важно, что в этот же слой входит общий visual resource system:

- `strings.xml` в `composeResources/values`;
- shared font resources;
- shared drawable resources;
- общие color и typography tokens.

То есть presentation layer здесь состоит не только из composable-функций, но и из общего набора UI-ресурсов, который используется всеми платформами.

UI построен по state-driven модели:

- пользователь нажимает кнопку;
- `ViewModel` обновляет state;
- Compose переотрисовывает нужные элементы.

## 5. DI layer через Koin

Koin здесь не отдельный "архитектурный этаж", а клей между слоями.

Он отвечает за:

- создание `ViewModel`;
- регистрацию storage/repository зависимостей;
- подключение сервисов к UI;
- единый способ резолва зависимостей в shared-коде.

Практически это значит, что экранный код не создаёт руками:

- HTTP client;
- storage implementation;
- repository;
- database wrapper.

Вместо этого эти зависимости собираются через DI-граф.

Для Aurora пришлось уделить отдельное внимание моменту старта Koin, поэтому в проекте появился Aurora-friendly bootstrap сценарий и отдельная диагностика статуса инициализации.

## 6. Data layer

Data layer отвечает за всё, что связано с получением, хранением и чтением данных.

Он состоит из нескольких частей:

- модели данных;
- network repository/service;
- persistence storage abstraction;
- Room-based реализация хранения;
- маппинг между persistence и UI/domain-моделями.

### Network часть

Network-слой делает следующее:

- создаёт HTTP client через shared API + platform engine;
- выполняет GET-запрос;
- сериализует JSON в модель;
- возвращает данные экрану.

UI не знает о конкретном transport backend. Он знает только, что можно выполнить "check Ktor" и получить результат.

### Persistence часть

Persistence-слой сейчас построен вокруг Room.

Он включает:

- entities;
- DAO;
- database definition;
- storage abstraction;
- операции сохранения и чтения поста;
- операции сохранения и чтения последней заметки.

Это позволяет демонстрировать сразу два сценария:

- технический persistence smoke test;
- пользовательский сценарий с текстовой заметкой, которая сохраняется между запусками.

На текущем этапе один и тот же Room-based persistence shape уже используется на всех активных target проекта:

- Android;
- iOS;
- Desktop;
- Aurora.

## 7. Поток данных в приложении

Ниже полезно описать не только слои, но и фактическое движение данных.

### Сценарий `Check Ktor`

Поток выглядит так:

1. пользователь нажимает кнопку;
2. экран или `ViewModel` запускает network action;
3. общий Ktor client выполняет запрос;
4. JSON десериализуется;
5. результат попадает в state;
6. UI показывает статус и данные;
7. при необходимости пост сохраняется в Room;
8. лог-панель обновляется.

### Сценарий `Save test post`

Поток выглядит так:

1. пользователь нажимает кнопку сохранения;
2. создаётся тестовая запись;
3. storage API вызывает Room;
4. запись сохраняется;
5. статус обновляется;
6. UI показывает подтверждение и лог.

### Сценарий текстовой заметки

Поток выглядит так:

1. пользователь вводит текст;
2. UI хранит input state;
3. по нажатию `Save` вызывается storage;
4. последняя заметка сохраняется в Room;
5. при чтении или повторном запуске она поднимается из БД;
6. UI показывает последнее сохранённое значение.

### Сценарий времени

Поток выглядит так:

1. общий `Flow` раз в секунду получает текущее время;
2. значение форматируется в `HH:mm:ss`;
3. UI подписывается на поток;
4. Compose выполняет локальную переотрисовку только нужной части экрана.

Это хороший пример того, что shared reactive UI реально работает на нескольких target.

## 8. Почему архитектура здесь именно такая

Можно было бы сделать PoC намного проще:

- без DI;
- без `ViewModel`;
- без persistence abstraction;
- с прямыми вызовами Ktor из composable;
- с прямым хранением состояния внутри одного файла.

Но это дало бы мало пользы для реального переноса настоящего проекта.

Текущая архитектура специально ближе к "боевому" приложению:

- UI отделён от data layer;
- storage спрятан за интерфейсом;
- зависимости инжектятся через DI;
- platform-specific слой остаётся тонким;
- shared-код содержит основную ценность приложения.

Поэтому результаты этого PoC можно экстраполировать на более крупный проект заметно легче.

## 9. Где в архитектуре был главный риск

Главный риск оказался не в Ktor и не в Room.

Он оказался в связке:

- Compose runtime;
- Aurora launcher/runtime lifecycle;
- повторный запуск приложения из GUI Aurora;
- стабильность сложного shared UI после restart.

Именно поэтому в архитектуру пришлось добавить не только "обычное приложение", но и диагностический режим.

С архитектурной точки зрения это означает:

- проект содержит normal app path;
- проект содержит smoke/diagnostic path;
- оба сценария используют один и тот же shared фундамент, но служат разным задачам.

## 10. Краткий итог по архитектуре

Архитектура проекта получилась такой:

- platform layer отвечает за запуск и интеграцию;
- shared bootstrap соединяет runtime и приложение;
- presentation layer живёт в `commonMain` и содержит основной UI;
- Koin склеивает зависимости;
- Ktor обеспечивает общий сетевой слой;
- Room обеспечивает общий persistence backend;
- Aurora-specific build/runtime слой адаптирует всё это к Aurora OS.

Главный результат в том, что shared-код здесь не декоративный, а действительно основной.

Платформенный код остался важным, но он в основном занимается интеграцией, а не переписыванием бизнес-логики и UI заново для каждой платформы.
